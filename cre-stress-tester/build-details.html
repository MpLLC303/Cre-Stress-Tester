How This Was Built

A look under the hood — the architecture, the prompt engineering, the tradeoffs, the testing protocol.

§ 01 — Architecture

A full-stack application with a curated knowledge base.

The tool is a single-page web application hosted on Vercel. The frontend is vanilla HTML, CSS, and JavaScript — no frameworks, no build step. The backend is a Node.js serverless function that receives the submitted memo, loads the six knowledge files, and calls the Anthropic Claude API with a production-hardened system prompt. The structured XML response is parsed and rendered into visual sections.

§ 02 — Prompt Engineering Stack

Ten proven techniques, applied across every prompt.

The system prompt governing the tool is built on a deliberate stack of ten prompt engineering techniques. Each one prevents a specific failure mode — hallucination, over-flagging, generic critique, sycophancy, response bloat. The production prompt combines seven of the ten in a single instruction set.

role priming xml delimiters output schemas few-shot examples chain-of-thought negative constraints self-critique loops confidence calibration length caps refusal anchors

The most important of these for this tool are refusal anchors and confidence calibration — these are what prevent the tool from making up market data or giving you the "yes this is a great deal" response that undermines stress-testing. Every time the tool doesn't know a specific number, it says so and points to a verifiable source instead of guessing.

§ 03 — Three-Version Iteration

The system prompt was rebuilt twice.

The production version (v3) is the result of two visible iteration steps. Each version was tested against the same four test cases to measure the improvement.

Version 01

Baseline

Role priming only. No knowledge files. Generic critique with frequent hallucination. Deliberately weak to establish a measurable starting point.

Version 02

Knowledge Files

Added the six curated knowledge files. Added a structured intake. Added negative constraints on marketing language. Material improvement in citation discipline.

Version 03

Production

Seven explicit rules, mandatory XML output schema, refusal anchors, chain-of-thought preamble, mandatory disclaimer. This is the version running in production.

§ 04 — Model Selection & Cost

Sonnet 4.6 vs. Haiku 4.5 — a real engineering tradeoff.

I benchmarked Claude Sonnet 4.6 against Claude Haiku 4.5 across the four-persona test set. Sonnet catches roughly 90% of embedded red flags; Haiku catches about 70% but responds in under 10 seconds. For the demo-heavy workload of a portfolio tool, Haiku's speed is the right tradeoff.

Prompt caching on the knowledge base means every call after the first gets a 90% discount on input tokens, because the six knowledge files are static across calls. The effective per-analysis cost is under two cents.

§ 05 — The Knowledge Base

Six curated files, roughly 10,000 tokens of grounded reference.

The knowledge base was assembled from public industry sources — CBRE Research, NAREIT T-Tracker, NCREIF, Green Street Advisors, NAIOP, and Adventures in CRE. Every numeric range in the files is flagged with a [VERIFY] tag and was sanity-checked against the published data before upload.

  1. Fundamentals Reference — pro forma structure, core return metrics, investment strategy definitions, capital stack basics.
  2. Red Flag Catalog — 25 named patterns across five categories: revenue, expense, cap rate & exit, debt & capital structure, structural & market.
  3. Asset Class Market Ranges — institutional underwriting ranges by asset class (multifamily, office, industrial, retail, hospitality).
  4. Sensitivity Framework — the hierarchy of assumption sensitivity and how to communicate fragility quantitatively.
  5. Sample Annotated Memos — one defensible memo and one weak memo, with red flags cited by name.
  6. Cap Rate & Cycle Reference — historical cap rate context and the Treasury-yield relationship.
§ 06 — Testing Protocol

Four test cases, five scoring dimensions, one control.

Four test personas were constructed with known answer keys — each a fictional but realistic deal memo with a specific number of embedded red flags. The tool was scored on five dimensions: red flag detection, citation discipline, fragility analysis, no-hallucination compliance, and analytical depth.

Persona C is a deliberate control case. It contains only four minor concerns, and the deal is mostly defensible. The test isn't just "can the tool catch red flags?" — it's also "can the tool recognize when there aren't any?" A tool that flags everything is as useless as one that flags nothing. This matters more than it sounds.

§ 07 — Honest Limitations

What this tool can't do.

Limitations

The tool cannot verify claims in the pitch document — if a memo says rent grew 5% last year, the tool cannot check whether that happened. It flags assumptions that look unrealistic against industry norms. The tool does not have live market data, so it refuses to quote current cap rates or loan spreads. It is not a substitute for due diligence. It is not investment advice. The quality of its critique is limited by the quality of the submitted memo.
